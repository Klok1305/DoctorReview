# Выпуск и обновление приложения

## Облачные обновления через GitHub Releases

Приложение получает обновления из публичных релизов
[`Klok1305/DoctorReview`](https://github.com/Klok1305/DoctorReview/releases). В репозитории
хранятся только исходники и файлы сборки. Установщики, `latest.yml` и `.blockmap`
публикуются как вложения релиза, а рабочие SQLite-базы, выгрузки 1С, Excel/PDF,
резервные копии и локальные настройки запрещены белым списком и проверкой CI.

Чтобы выпустить новую версию:

1. Изменить `version` в `package.json`, например с `1.0.6` на `1.0.7`.
2. Закоммитить и отправить изменения в ветку `main`.
3. Создать и отправить тег с той же версией:

   ```powershell
   git tag -a v1.0.7 -m "DoctorReview 1.0.7"
   git push origin v1.0.7
   ```

4. Дождаться успешного задания «Выпуск обновления» в GitHub Actions.

Workflow на Windows проверяет, что тег совпадает с `package.json`, устанавливает
зависимости по lock-файлу, запускает аудит репозитория и тесты, затем собирает NSIS
и portable-версию. Для загрузки в GitHub Releases используется встроенный
`GITHUB_TOKEN`; отдельный токен добавлять не нужно.

Установленная NSIS-версия проверяет адрес
`https://github.com/Klok1305/DoctorReview/releases/latest/download`, скачивает
обновление в фоне и предлагает установить его. Перед установкой приложение делает
резервную SQLite-копию. Рабочая папка и база находятся вне каталога программы и
обновлением не заменяются.

Версия 1.0.5 — переходная: её нужно один раз установить вручную поверх 1.0.3.
Начиная со следующего релиза обновления будут находиться автоматически.

## Тест обновления 1.0.5 → 1.0.6

Для проверки используйте именно установленную NSIS-версию `1.0.5`. Portable-файл
обновляется заменой вручную и не подходит для проверки автообновления.

### 1. Подготовить старую версию

1. Скачайте и установите `DoctorReview-Setup-1.0.5-x64.exe` из релиза `v1.0.5`.
2. Запустите приложение и на вкладке «Данные» убедитесь, что в блоке «Рабочее место»
   показана версия `1.0.5`.
3. Добавьте небольшую тестовую запись или запомните существующие данные — после
   обновления они должны сохраниться.

### 2. Опубликовать тестовую версию 1.0.6

В версии `1.0.6` слово «работы» в верхнем заголовке стало нежно-салатовым, а
«врачей» — фиолетовым. Это визуальный признак успешного обновления.

```powershell
pnpm run assemble
pnpm run audit:repo
pnpm test
git add .github/workflows/release.yml README.md build/app.css build/index.template.html `
  docs/ОБНОВЛЕНИЕ.md index.html package.json tests/build.test.cjs `
  tests/desktop-services.test.cjs
git commit -m "test: verify cloud update"
git push origin main
git tag -a v1.0.6 -m "DoctorReview 1.0.6"
git push origin v1.0.6
```

Откройте на GitHub вкладку **Actions** и дождитесь зелёного результата workflow
«Выпуск обновления». Затем проверьте релиз `v1.0.6`: в нём должны быть установщик,
portable-файл, `.blockmap` и `latest.yml`. В `latest.yml` должно быть `version: 1.0.6`.

### 3. Проверить обновление в приложении

1. Откройте установленную версию `1.0.5` и перейдите на вкладку «Данные».
2. В блоке «Рабочее место» нажмите «Проверить обновления». Можно также подождать
   около 15 секунд после запуска — автоматическая проверка выполняется сама.
3. Дождитесь сообщения о готовой версии `1.0.6`. Кнопка изменится на
   «Установить загруженное».
4. Нажмите эту кнопку. Приложение создаст резервную копию базы, закроется, запустит
   установщик и снова откроется после обновления.
5. Убедитесь, что указана версия `1.0.6`, слово «работы» стало салатовым, слово
   «врачей» — фиолетовым, а тестовые данные остались на месте.
6. Проверьте папку «Резервные копии»: в ней должна появиться копия, созданная перед
   обновлением.

Если приложение сообщает, что обновлений нет, откройте
`https://github.com/Klok1305/DoctorReview/releases/latest/download/latest.yml` и
проверьте, что GitHub уже отдаёт версию `1.0.6`, а не предыдущий релиз.

## Ручная и portable-установка

Если автоматическое обновление недоступно, скачайте NSIS-файл
`DoctorReview-Setup-<версия>-x64.exe` из раздела Releases. В старой версии можно нажать
«Установить из файла» и выбрать этот установщик.

Portable-версия обновляется заменой старого portable `.exe` новым. Данные остаются
в отдельно выбранной рабочей папке.

## Локальная проверка перед релизом

```powershell
pnpm run audit:repo
pnpm test
.\scripts\build-desktop.ps1 -SmokeTest
```

Готовые локальные файлы появляются в `dist`, который игнорируется Git.

## Подпись Windows

Без сертификата Windows SmartScreen может показать предупреждение. Когда появится
code-signing сертификат, добавьте в GitHub Actions secrets `WIN_CSC_LINK` и
`WIN_CSC_KEY_PASSWORD`. Сам `.pfx` запрещено добавлять в Git.

## Совместимость базы

Версия схемы записывается в `schema_migrations`. Изменения схемы добавляются новой
миграцией; существующие миграции не переписываются. Перед установкой обновления
создаётся резервная копия текущей базы.
